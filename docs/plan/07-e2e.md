# Step 7 — End-to-End Validation & Release

## Objective

Prove the whole pipeline on a realistic review and package v0.1.0.

## Deliverables

- [x] `examples/srs-demo/` — fictional SRS (~10 sections) + scripted
      3-reviewer scenario (typos, duplicates, technical majors, one freeze
      violation, one reopen) — fully synthetic content
- [x] Scripted E2E test driving init → freeze → copies → harvest → triage →
      disposition (headless rtd.yaml edits) → implement → verify → finalize
- [x] `docs/usage.md` — user guide for both modes (human owner / AI owner)
- [x] README completed (install, quickstart, GIF/screenshot of GUI)
- [x] Packaging: installable via `pipx install .`; version 0.1.0; tag `v0.1.0`

## Definition of Done

A newcomer can run the demo review start-to-finish following `docs/usage.md`
only; E2E test green in CI-less local run (`python -m pytest -q`); tagged
release pushed.

## Deviations

- **README GUI image:** the GUI is described and linked, but no static
  screenshot/GIF is embedded — a headless environment can't persist an image into
  the repo. `gui/rtd.html` opens by double-click; the visual pass lives in
  `docs/spec/gui-test-checklist.md`.
- **Prompts moved into the package** (`src/malus/prompts/`, loaded via
  `importlib.resources`, shipped as package data) so `malus ai` works from a plain
  `pip install .` / `pipx install .` — refines the Step-6 repo-root location.
  Verified in a clean non-editable venv.
- **E2E disposition** is headless `rtd.yaml` edits (as the plan specifies),
  simulating the owner/GUI; owner-side status advances are headless too, while the
  reviewer-side (verify / reopen) and finalize go through the enforced functions.
- **Quickstart** is run from the maluS repository root (captures `$MALUS`) because
  the example ships in the repo, not with the installed package.
- **`v0.1.0` tag** is created and pushed at the release step (after confirmation).
