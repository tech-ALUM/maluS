# v1.7 Step 1 — AI Co-Owner Drafted Dispositions Decisions

Requested by Alberto (2026-07-15): Claude as a review co-owner reads the
reviewers' comments over the API, discusses them with a human in chat, then
uploads a distilled disposition — which a human owner confirms in the GUI before
it takes effect.

## Observations

- [decision] The AI takes the OWNER ROLE as a co-owner (assignable from the existing members GUI); the human primary owner (Review.owner_id) is unchanged, so a confirmed disposition stays attributed to the human. No ownership-transfer feature, no new role, no new Status, no migration — `ai_drafted` (already on the RID, previously unused) is the flag #model
- [decision] A DRAFT = a RID still OPEN with disposition(+reply/resolution) set and ai_drafted=True. ai_drafted is auto-set in `update_rid` when the actor is_ai. Confirm = the existing dispose → `answer` transition (ai_drafted retained as provenance). Discard = a new `discard_disposition_draft` service clearing the fields + ai_drafted back to a plain OPEN #draft
- [decision] is_ai COMMIT GUARD: an AI principal may draft but never commit an owner decision. Enforced authoritatively in the services (`answer`/`implement`/`finalize` raise ClosureAuthorityError → 403 when by.is_ai), covering API + GUI; the API/GUI also call `authz.forbid_ai_commit` at the boundary (patch_rid status branch, finalize route, discard route) #guard #authz
- [decision] The guard also covers CONTENT mutations (hardened right after v1.7 shipped): `freeze_baseline`, `save_version` (→ set-document + create-change), and `apply_suggestions` refuse an AI actor (403). A self-review found these endpoints were reachable by an AI co-owner, contradicting the v1.7 out-of-scope ("AI-owner does not freeze/implement/… — human owner only"). Now an AI co-owner can ONLY draft a disposition — it cannot freeze, replace the document, create a version, or apply SUGG edits #guard #hardening
- [context] This does NOT touch the closure invariant: disposition is an owner power, closure (verify/reopen) stays reviewer-side and the existing is_ai verify block is unchanged — so humans keep the final say by closing #invariant
- [decision] MCP tool `submit_disposition(review_id, rid, disposition, reply, resolution)` PATCHes the fields WITHOUT a status → always a draft; added to TOOL_NAMES + build_server. The module still exposes no verify/close tool #mcp
- [decision] GUI: the human owner sees an "AI proposal" banner on the finding page with Confirm (the existing dispose form, pre-filled) and Discard; shown only to a human owner (can_dispose = owner AND not is_ai). RTD-row "AI" badge + a dashboard "AI proposals" tile counting OPEN ai_drafted RIDs #gui
- [context] No live-browser verification: v1.7 adds no novel client-side behavior (Confirm/Discard are single-button hx-boost forms — the proven pattern; the v1.6 bug was specific to TWO submit buttons). Covered by pytest (rendered markup + behavior). Full suite green (246) #testing
