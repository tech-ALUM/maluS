# v1.3 Step 1 — Admin Hard-Delete of a User Decisions

Requested by Alberto (2026-07-12): the admin could only deactivate (soft) a
user. v1.3 adds a true hard delete (erasure) from the admin GUI, with
reassignment/anonymization so referential integrity and the review/audit history
survive.

## Observations

- [decision] A hard delete reassigns every reference before removing the row: reviews the target PRIMARY-OWNS (Review.owner_id, NOT NULL) go to a new owner the admin chooses per review; findings/verifications/versions/audit go to a shared login-less sentinel 'Deleted user'; the target's memberships and reviewer copies are deleted #delete
- [decision] Single SHARED sentinel (username `deleted-user`, is_active=False), not a per-deleted-user tombstone — the person is erased, the records are kept; every deleted person shows as 'Deleted user' in the RTD/audit. Lives in services/core.py (SENTINEL_USERNAME + sentinel_user()), created lazily on first delete #delete
- [decision] GUI is a 2-step admin flow: GET /ui/admin/users/{username}/delete (confirm page listing owned reviews, each with a new-owner picker over active non-target accounts) then POST with owner_for_<review_id_str> per owned review; the POST is async and reads the dynamic fields via request.form() #gui
- [decision] Guards (server-side): admin-only (403); cannot delete yourself or the sentinel (409); every owned review must get a valid new owner (existing, active, != target) else 422 and nothing is deleted #guard
- [decision] AUDIT preserved + anonymized (actor -> sentinel), not purged — per Alberto: hard-delete the person without holes in the review history. Deactivate stays as the reversible soft option #audit
- [context] delete_user is a services/core.py function (transactional; reassign then delete then log). No schema change / no Alembic migration (data only). No JavaScript added #impl
- [context] Tests: tests/db/test_delete_user.py (service reassignment matrix incl. verified_by/created_by/audit + owned-review transfer) and tests/web/test_admin_delete.py (guards, owner-picker flow, end-to-end anonymization, sentinel hidden/undeletable). Full suite green (209). Import note: the db test creates users via UserRepo.get_or_create, not auth.service.create_user, to avoid the auth<->api circular import when api isn't preloaded #testing
