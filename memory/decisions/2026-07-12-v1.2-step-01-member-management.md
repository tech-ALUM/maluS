# v1.2 Step 1 — Member Management (picker, roles, removal) Decisions

Requested by Alberto (2026-07-11 design, 2026-07-12 implementation): the browser
already assigned per-review roles, but by a free-text display name that spawned
phantom users on a typo, with no search, no inline role change, and no removal.

## Observations

- [decision] The Members page now assigns EXISTING accounts by stable `username` via a searchable HTMX typeahead; `get_or_create` is removed from the GUI path so a typo can no longer spawn a phantom user. An unknown or inactive username is rejected 422 #picker
- [decision] Candidate search is review-scoped: `GET /ui/reviews/{id}/members/search?q=` returns an HTML fragment of active, non-member accounts, owner/admin-gated, capped at 20; a small inline `malusPickCandidate()` fills a hidden `username`. This deviates from the spec's global `/ui/accounts/search` so it can gate by the review and exclude that review's existing members #picker
- [decision] Inline role change reuses `POST /ui/reviews/{id}/members` (the `set_member_role` upsert) via an auto-submitting `<select>`; member removal is a new `POST /ui/reviews/{id}/members/{username}/remove` backed by `ReviewRepo.remove_member` #members
- [decision] Owner-safety guard: the primary owner (`Review.owner_id`) can be neither demoted nor removed (409); since the primary is always an owner, this alone guarantees >=1 owner. Ownership transfer is out of scope. Removing a member leaves their harvested RIDs intact (no cascade) #invariant
- [context] The owner is dual-represented — `Review.owner_id` (drives disposition attribution) plus a `ReviewMember(role=owner)` (what `authz.review_role` reads for RBAC); `create_review` seeds both. The guard keys on `owner_id` #model
- [context] At Step 1 the JSON API `POST /reviews/{id}/reviewers` was left unchanged (`name` + `get_or_create`); the GUI Members page was the step's scope. SUPERSEDED in the v1.2.0 release: the endpoint now requires an existing active account and rejects an unknown/inactive name (422), mirroring the GUI — while bulk `create_review` and the legacy `import` path stay as-is (import materializes placeholder users by design). See CHANGELOG v1.2.0 and the `fix(api)` commit #scope
- [context] 13 new tests in tests/web/test_members.py (assign-by-username, anti-phantom, search filter/gate, role change, removal, primary-owner guard, RID persistence, template wiring); full suite green (196). Typeahead + assign + remove live-verified in a browser (picker JS confirmed; the screenshot tool timed out, so the accessibility snapshot was used as proof) #testing
