# v1.8 Step 1 — Reviewer Comment Retraction + Submissions Panel Placement Decisions

Requested by Alberto (2026-07-16): deleting a comment from the editor must also
remove it from the RTD table; a reviewer must be able to delete their own comments
from the table too; and the submissions panel is badly placed — move it to the top.

## Observations

- [decision] A reviewer retracting their own comment HARD-DELETES it, reconciled with the deliberate "vanished → withdrawn, not deleted" harvest decision as: pristine → delete, acted-upon → keep withdrawn. Pristine = the owner never engaged (no disposition/reply/resolution, not verified, no RidChange, not in a cluster). This preserves the trace of anything the owner already disposed #retraction
- [decision] The pure harvest core (build_rtd) is UNCHANGED — it still withdraws (test_vanished_comment_becomes_withdrawn_not_deleted stays green). The hard-delete is a DB-service cleanup `_purge_retracted` run at the end of the `harvest` service, so every harvest (reviewer save, API, moderator) reconciles #harvest #core
- [decision] Both delete paths funnel through the reviewer's COPY (the single source of truth). Editor: removing the block + saving already vanishes it → purge. Table: new `retract_comment` service parses the copy (parser.scan), matches the block to the RID by identity (`_block_matches` + `_sugg_repr`, mirroring harvest._render_sugg), cuts its span, re-saves (preserving submitted_at), harvests → purge. The table control never bypasses the copy, so a later harvest can't resurrect the comment #retraction #copy
- [decision] Table retract is reviewer-only, own comment, OPEN only: route `POST /ui/reviews/{id}/rids/{rid}/retract` → 403 if not the RID's own reviewer, 409 if not OPEN. GUI: a small "✕ delete" control shown only on the current reviewer's own OPEN rows #authz #gui
- [decision] The v1.6 submissions panel moved from the right-hand two-column `.rtd-layout` to a full-width band at the TOP (below header/actions, above metrics/table); reviewers as a wrapping row of state chips #gui
- [context] Live verification note: the preview renderer reports innerWidth 0 (collapses every column and times out screenshots); verified the layout after forcing a 1280x800 viewport (panel full-width above the table, chips in one row) and exercised retract end-to-end (own OPEN → row hard-deleted). No schema change / no migration. Full suite green (255) #testing
